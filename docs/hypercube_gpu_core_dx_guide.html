Comment ça marche — en 30 secondes

Le core suit toujours le même chemin. Comprendre ce flux, c'est comprendre 80% de la lib.

1. Manifest JSON
→

Tu déclares la mémoire (faces), la physique (rules), la topologie (grid)

2. createSimulation()
→

Valide le manifest, alloue la VRAM et construit les objets GPU

3. engine.step(kernels)
→

Dispatch les compute shaders WGSL, swap parity

4. syncFacesToHost
→

Rapatrie seulement les faces nécessaires (jamais toute la VRAM)

5. getInnerFaceData
→

Extrait les données sans les ghost cells — prêtes à rendre

La donnée vit en VRAM du premier au dernier step. Le CPU ne touche à rien pendant le calcul.

Trois objets à connaître

HypercubeManifest Le JSON de config complet (engine + config)
GpuEngine L'objet principal — step(), syncFacesToHost(), getInnerFaceData()
SimulationBuilder Fluent API pour construire le manifest sans écrire le JSON à la main

Les Faces — la mémoire de ta simulation

Une face = une couche de données en VRAM. Chaque cellule de la grille a une valeur par face. Clique sur un type pour voir ce qu'il génère.

Quand utiliser pingPong ?

pingPong: false Donnée lue et écrite dans le même buffer. OK pour obstacles, masques, macros dérivées (rho, ux).
pingPong: true Double buffer auto. Le GPU lit "Now" et écrit "Next". Obligatoire quand une cellule dépend de ses voisines (LBM, wave, diffusion).

numComponents — pour les faces multi-scalaires

{ "name": "f", "type": "population", "numComponents": 9 }
// D2Q9 → 9 directions → 9 floats par cellule
// D3Q19 → 19 composants → "numComponents": 19
// Omis = déduit du type : scalar=1, vector=3, population=9

numSlots — pour le triple buffering (wave)

{ "name": "u", "type": "field", "numSlots": 3 }
// Wave du 2nd ordre : besoin de t-1, t, t+1
// numSlots: 3 → ParityManager gère le cycle modulo 3

WGSL et les macros auto-injectées

Quand tu appelles engine.getWgslHeader('collision'), le core injecte automatiquement des helpers pour toutes tes faces. Tu n'écris jamais d'index manuels.

Le header injecté — structure

// Automatiquement injecté avant ton code :

struct Uniforms {
  nx: u32,         // colonnes (sans ghost cells)
  ny: u32,         // lignes (sans ghost cells)
  strideRow: u32,  // nx + 2*ghostCells
  physicalNy: u32, // ny + 2*ghostCells
  t: u32,          // tick courant (frame)
  tick: u32,       // même chose (alias)
  p0..p7: f32,     // tes params (tau=p0, ux0=p1...)
  ...
}
@group(0) @binding(1) var<uniform> uniforms: Uniforms;

// Helpers d'indexation :
fn getIndex(x: u32, y: u32) -> u32
fn getIndex3D(x: u32, y: u32, z: u32) -> u32

// Pour chaque face déclarée — ex: face "rho" scalar :
fn read_rho(x: u32, y: u32) -> f32
fn write_rho(x: u32, y: u32, val: f32)

// Pour face pingPong — ex: face "f" population :
fn read_f_Now(x: u32, y: u32, d: u32) -> f32
fn write_f_Next(x: u32, y: u32, d: u32, val: f32)

Correspondance params → WGSL

// Dans le manifest :
"params": { "tau": 0.55, "ux0": 0.08, "reynolds": 1000 }

// Dans le WGSL :
let omega = 1.0 / uniforms.p0;   // tau → p0 (ordre de déclaration)
let ux0   = uniforms.p1;          // ux0 → p1
let re    = uniforms.p2;          // reynolds → p2
Les params sont mappés par ordre dans params, pas par nom. tau est toujours p0 parce qu'il est déclaré en premier.

Kernel minimal complet

const lbmSource = engine.getWgslHeader('collision') + `

@compute @workgroup_size(16, 16)
fn main(@builtin(global_invocation_id) id: vec3<u32>) {
  let x = id.x; let y = id.y;
  if (x >= uniforms.nx || y >= uniforms.ny) { return; }

  // Init au tick 0
  if (uniforms.tick == 0u) {
    write_rho_Now(x, y, 1.0);
    write_ux_Now(x, y, uniforms.p1);
    for (var d = 0u; d < 9u; d++) {
      write_f_Next(x, y, d, lbm_feq(1.0, uniforms.p1, 0.0, d));
    }
    return;
  }

  // Obstacle bounce-back
  if (read_obs(x, y) > 0.5) {
    for (var d = 0u; d < 9u; d++) {
      write_f_Next(x, y, d, read_f_Now(x, y, OPP[d]));
    }
    return;
  }

  // ... collision BGK
}`;

await engine.step({ 'collision': lbmSource });

Lire les résultats

Trois patterns selon ton besoin. Le premier ne coûte rien, le dernier coûte beaucoup — mais parfois nécessaire.

Pattern A — rendu direct (zéro coût)

Le buffer GPU est utilisé directement dans ton renderer. Aucun transfert CPU/GPU.

// Three.js : lier le buffer VRAM à un shader de rendu
const storageBuffer = engine.buffer.gpuBuffer;

const bindGroup = device.createBindGroup({
  layout: renderPipeline.getBindGroupLayout(0),
  entries: [{ binding: 0, resource: { buffer: storageBuffer } }]
});
// → ton vertex shader lit rho/ux/uy directement en VRAM

Pattern B — sync partielle (diagnostic, UI)

Rapatrie seulement les faces nécessaires. Ne bloque pas le render loop.

// À appeler hors du render loop (setTimeout, 1×/seconde)
async function updateHUD() {
  // Sync seulement 'rho' — pas toute la VRAM
  await engine.syncFacesToHost(['rho']);

  // Lecture SANS ghost cells (prête à itérer)
  const rho = engine.getInnerFaceData('chunk_0_0_0', 'rho');

  const avg = rho.reduce((a, b) => a + b, 0) / rho.length;
  document.getElementById('density').textContent = avg.toFixed(4);
}
setTimeout(updateHUD, 1000);

// Convention : chunk_x_y_z (position dans la grille de chunks)

Pour un aquarium 256×128 en 4×2 chunks → la grille physique fait 64×64 cellules par chunk. getInnerFaceData te retourne un Float32Array de 64×64 = 4096 valeurs par chunk.

La boucle principale

await engine.ready();

const kernels = {
  'lbm':     engine.getWgslHeader('lbm')     + lbmSource,
  'poisson': engine.getWgslHeader('poisson') + poissonSource,
  'repulse': engine.getWgslHeader('repulse') + repulseSource,
  'agent':   engine.getWgslHeader('agent')   + agentSource,
  'tensor':  engine.getWgslHeader('tensor')  + tensorSource,
};

async function loop() {
  await engine.step(kernels, 1);  // 1 tick = 5 passes GPU
  requestAnimationFrame(loop);
}
loop();

// Diagnostic 1×/seconde — seulement ux/uy pour le rendu UI
