HYPERCUBE

Universal Registry

Architecture & Flux de Données

Le Core d'Hypercube suit un pipeline déterministe pour garantir une performance maximale en VRAM.

1. Manifest JSON
Déclaration de la mémoire (Faces), de la physique (Rules) et de la topologie (Grid).
2. createSimulation()
Validation du contrat de données, allocation VRAM et compilation des kernels.
3. engine.step()
Exécution des Compute Shaders. Le Dispatcher gère automatiquement la parité, les offsets et les atomiques.
Zéro Readback : La donnée ne quitte jamais la VRAM pendant la simulation.

Composants Fondamentaux

GpuDispatcher Moteur de génération de macros WGSL (Standard + Atomic).
DataContract Garant de l'intégrité de la mémoire et du mapping des faces.
VirtualGrid Abstraction topologique (2D/3D) découpée en Chunks physiques.
MasterBuffer Gestionnaire de dual-buffer (Standard/Atomic) en VRAM.

SimulationPacks.LBM_D2Q9, .WAVE_2D, .DIFFUSION_2D, .CELLULAR_AUTOMATA — démarrage en 3 lignes.
Le champ config.engine (string) doit être identique à engine.name — c'est la clé de liaison. Un mismatch est silencieux.

Les Faces — la mémoire de ta simulation

Une face = une couche de données en VRAM. Chaque cellule de la grille a une valeur par face.

Quand utiliser pingPong ?

pingPong: false Donnée lue et écrite dans le même buffer. OK pour obstacles, masques, atomiques.
pingPong: true Double buffer auto. Le GPU lit "Now" et écrit "Next". Obligatoire quand une cellule dépend de ses voisines (LBM, wave, diffusion).
Les faces atomiques héritent nativement de pingPong: false pour permettre l'accumulation inter-threads.

WGSL et les macros auto-injectées

Quand tu appelles engine.getWgslHeader('collision'), le core injecte automatiquement des helpers pour toutes tes faces. Tu n'écris jamais d'index manuels.

Le header injecté — structure

// Automatiquement injecté avant ton code :

struct Uniforms {
  nx: u32, ny: u32, strideRow: u32, physicalNy: u32,
  t: u32, tick: u32,
  p0..p7: f32,
  ...
}
@group(0) @binding(0) var<storage, read_write> data: array<f32>;
@group(0) @binding(1) var<uniform> uniforms: Uniforms;
@group(0) @binding(2) var<storage, read_write> dataAtomic: array<atomic<u32>>;

// Pour face atomique (Ex: 'flux' type 'atomic_f32') :
fn atomicAdd_flux(x: u32, y: u32, val: f32)
fn atomicLoad_flux(x: u32, y: u32) -> f32

// Pour face standard (Ex: 'phi') :
fn read_phi(x: u32, y: u32) -> f32
fn write_phi(x: u32, y: u32, val: f32)
v5.0.4 : Le support atomicAdd pour les floats est géré par une boucle CAS interne invisible pour l'utilisateur.

Correspondance params → WGSL

// Dans le manifest :
"params": { "tau": 0.55, "ux0": 0.08 }

// Dans le WGSL :
let viscosity = uniforms.p0;   // tau
let velocity  = uniforms.p1;   // ux0

Lire les résultats

Trois patterns selon ton besoin. Le premier ne coûte rien, le dernier coûte beaucoup — mais parfois nécessaire.

Pattern A — rendu direct (zéro coût)

Le buffer GPU est utilisé directement dans ton renderer. Aucun transfert CPU/GPU.

// Three.js : lier le buffer VRAM à un shader de rendu
const storageBuffer = engine.buffer.gpuBuffer;

const bindGroup = device.createBindGroup({
  layout: renderPipeline.getBindGroupLayout(0),
  entries: [{ binding: 0, resource: { buffer: storageBuffer } }]
});

Pattern B — sync partielle (diagnostic, UI)

Rapatrie seulement les faces nécessaires. Ne bloque pas le render loop.

async function updateHUD() {
  // Sync seulement 'rho'
  await engine.syncFacesToHost(['rho']);
  const rho = engine.getInnerFaceData('chunk_0_0_0', 'rho');
  document.getElementById('density').textContent = rho[0].toFixed(4);
}
setTimeout(updateHUD, 1000);
L'accès aux faces atomiques via getInnerFaceData est identique aux faces standard. Le framework gère le remapping CPU.

Exemple complet — Universal Core

Le Core combine plusieurs systèmes qui partagent le même MasterBuffer. Voici comment le structurer.

La boucle principale

const kernels = {
  'compute': engine.getWgslHeader('compute') + computeSource
};

async function loop() {
  await engine.step(kernels);
  requestAnimationFrame(loop);
}
loop();

Power User Extensions (v5.0.4)

Fonctionnalités avancées pour le couplage multi-physique et le pilotage temps réel.

1. Floating Point Atomics
Accumulation de forces ou d'énergie par face via Binding 2. Injecte un CAS Loop automatique.
atomicAdd_face(x, y, val)
2. stepWithParams
Override dynamique des paramètres p0..p7 à chaque step pour le contrôle externe (PID, UI). 
engine.step(kernels, { tau: 0.8 })
