Architecture & Data Flow

The Hypercube Core follows a deterministic pipeline to ensure maximum performance within VRAM.

1. Manifest JSON
Declaration of memory (Faces), physics (Rules), and topology (Grid).
2. createSimulation(manifest)
Manifest validation via ManifestValidator, VRAM allocation (Standard + Atomic + Global), returning a ready GpuEngine.
3. engine.use(kernels)
Persistent registration of WGSL kernels. Pre-compiles GPU pipelines once. To be called before the simulation loop.
4. engine.step(N) & engine.params
Execution of N GPU ticks. The Dispatcher uses pre-compiled pipelines — zero WGSL overhead in the hot path. Parameters are dynamically modifiable via the engine.params Proxy.
Zero Readback: Data never leaves VRAM during simulation. Bindings: 0 = Standard (f32), 2 = Atomic (i32), 3 = Global Workspace.

Core Components

GpuDispatcher Dispatch orchestrator. Delegates to PipelineCache, UniformStagingManager, HaloExchangePass, and ReductionPass.
WgslHeaderGenerator Generates WGSL macros (Standard + Atomic + Global) from the DataContract.
DataContract Guardian of memory integrity. Exposes FaceMappings and GlobalMappings.
VirtualGrid Topological abstraction (2D/3D) partitioned into physical Chunks.
MasterBuffer Triple GPU allocation: gpuBuffer (binding 0), gpuAtomicBuffer (binding 2), and gpuGlobalBuffer (binding 3).
ManifestValidator Validates config.engine / engine.name linkage, face duplicates, and dimensional consistency.

Faces — Your Simulation's Memory

A face = a data layer in VRAM. Each grid cell has one value per face. Click on a type to see the generated macros.

When to use pingPong?

pingPong: false Data is read and written in the same buffer. Suitable for obstacles, masks, derived macros (rho, ux), and atomic faces.
pingPong: true Automatic double buffering. The GPU reads "Now" and writes "Next". Mandatory when a cell depends on its neighbors (LBM, wave, diffusion).
atomic_u32 and atomic_f32 faces reside in Binding 2 (dataAtomic), separate from the standard Binding 0. The framework manages allocation and binding automatically.

numComponents — Multi-Scalar Faces

Injected Header Structure

// Canonical Bindings (UniformLayout.ts — single source of truth)
@group(0) @binding(0) var<storage, read_write> data:       array<f32>;
@group(0) @binding(1) var<uniform>             uniforms:   Uniforms;
@group(0) @binding(2) var<storage, read_write> dataAtomic: array<atomic<i32>>;  // if using atomic faces
@group(0) @binding(3) var<storage, read_write> globals:    array<atomic<i32>>;  // if globals are defined

struct Uniforms {
  nx: u32, ny: u32, strideRow: u32, physicalNy: u32,
  t: f32,  tick: u32, strideFace: u32, numFaces: u32,
  p0: f32, p1: f32, p2: f32, p3: f32,
  p4: f32, p5: f32, p6: f32, p7: f32,
  faces: array<u32, 64>,
  ghosts: u32, leftRole: u32, rightRole: u32,
  topRole: u32, bottomRole: u32, frontRole: u32, backRole: u32
};

// Indexing Helpers
fn getIndex(x: u32, y: u32) -> u32
fn getIndex3D(x: u32, y: u32, z: u32) -> u32

// For pingPong standard face (e.g., "f" population):
fn read_f_Now(x: u32, y: u32, d: u32) -> f32
fn write_f_Next(x: u32, y: u32, d: u32, val: f32)

// For non-pingPong standard face (e.g., "rho" scalar):
fn read_rho_Now(x: u32, y: u32) -> f32
fn write_rho_Now(x: u32, y: u32, val: f32)

// For atomic_f32 face (e.g., "e_sum"):
fn atomicAdd_e_sum_Now(x: u32, y: u32, val: f32)   // Internal CAS loop
fn atomicLoad_e_sum_Now(x: u32, y: u32) -> f32

// For atomic_u32 face (e.g., "counter"):
fn atomicAdd_counter_Now(x: u32, y: u32, val: u32) -> u32
fn atomicLoad_counter_Now(x: u32, y: u32) -> u32
fn atomicStore_counter_Now(x: u32, y: u32, val: u32)

// For globals (e.g., "forces" count=2):
fn atomicAdd_global_forces(idx: u32, val: f32)  // idx=0 for Fx, idx=1 for Fy

Param-to-WGSL Mapping

// In Manifest:
"params": { "tau": 0.55, "ux0": 0.08, "reynolds": 1000.0 }

// In WGSL — mapped by declaration order:
let omega = 1.0 / uniforms.p0;   // tau    → p0
let ux0   = uniforms.p1;          // ux0    → p1
let re    = uniforms.p2;          // reynolds → p2

// In TypeScript — Semantic control via Proxy:
engine.params.tau = 0.6;              // Immediate update
engine.params.set('ux0', 0.1);        // Fluent API
await engine.step(1);                 // New values are active
Parameters are mapped by declaration order in config.params, not by name. tau is always p0 because it was declared first.

Data Extraction

Three patterns depending on your requirements. The first costs nothing; the last is computationally expensive but necessary for some physics audits.

Pattern A — getFace() — Recommended Helper

Synchronizes a face, strips ghost cells, and returns a clean Float32Array in a single line.

// Automatic Sync + Ghost Cell Stripping
const rho = await engine.getFace('rho');
console.log('Center Density:', rho[Math.floor(nx/2) + Math.floor(ny/2) * nx]);

Pattern B — Direct Rendering (Zero-Cost)

The GPU buffer is utilized directly in your renderer. No CPU/GPU transfer overhead.

// Three.js / Babylon.js / Custom WebGPU renderer
const storageBuffer = engine.buffer.gpuBuffer;  // Binding 0 — Standard faces

const bindGroup = device.createBindGroup({
  layout: renderPipeline.getBindGroupLayout(0),
  entries: [{ binding: 0, resource: { buffer: storageBuffer } }]
});

Pattern C — GPU Reduction (Energy, Forces)

Summation or accumulation performed directly on the GPU — zero readback, zero CPU summation drift.

// Scalar face reduction — result on GPU-side
const totalMass = await engine.reduceField('rho', ReductionCoreSource);

// Global Retrieval (e.g., aerodynamic forces via atomicAdd in WGSL)
const forces = await engine.getGlobal('forces');
const Fx = (forces[0] as number) / scaleFactor;
const Fy = (forces[1] as number) / scaleFactor;

Pattern D — Partial Sync (Diagnostics, UI)

// To be called outside the render loop — max 1x/second
async function updateHUD() {
  await engine.syncFacesToHost(['rho']);
  const rho = engine.getInnerFaceData('chunk_0_0_0', 'rho');
  const avg = rho.reduce((a, b) => a + b, 0) / rho.length;
  document.getElementById('density').textContent = avg.toFixed(4);
}
setInterval(updateHUD, 1000);
chunkId Convention: chunk_x_y_z. A 4×2 grid results in chunk_0_0_0 through chunk_3_1_0. A 1×1 grid always uses chunk_0_0_0.
getInnerFaceData() utilizes an internal buffer pool — zero GC pressure at 60fps. The pool key is chunkId:faceName:nx:ny:nz.
