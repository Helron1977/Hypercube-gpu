HYPERCUBE

Universal Registry

Architecture & Flux de Données

Le Core d'Hypercube suit un pipeline déterministe pour garantir une performance maximale en VRAM.

1. Manifest JSON
Déclaration de la mémoire (Faces), de la physique (Rules) et de la topologie (Grid).
2. createSimulation()
Validation du contrat de données, allocation VRAM et compilation des kernels.
3. engine.step()
Exécution des Compute Shaders. Le Dispatcher gère automatiquement la parité et les offsets.
Zéro Readback : La donnée ne quitte jamais la VRAM pendant la simulation.

Composants Fondamentaux

GpuDispatcher Moteur de génération de macros WGSL et gestionnaire de dispatch.
DataContract Garant de l'intégrité de la mémoire et du mapping des faces.
VirtualGrid Abstraction topologique (2D/3D) découpée en Chunks physiques.

Le champ config.engine (string) doit être identique à engine.name — c'est la clé de liaison. Un mismatch est silencieux.

Les Faces — la mémoire de ta simulation

Une face = une couche de données en VRAM. Chaque cellule de la grille a une valeur par face. Clique sur un type pour voir ce qu'il génère.

Quand utiliser pingPong ?

pingPong: false Donnée lue et écrite dans le même buffer. OK pour obstacles, masques, macros dérivées (rho, ux).
pingPong: true Double buffer auto. Le GPU lit "Now" et écrit "Next". Obligatoire quand une cellule dépend de ses voisines (LBM, wave, diffusion).

numComponents — pour les faces multi-scalaires

{ "name": "f", "type": "population", "numComponents": 9 }
// D2Q9 → 9 directions → 9 floats par cellule
// D3Q19 → 19 composants → "numComponents": 19
// Omis = déduit du type : scalar=1, vector=3, population=9

numSlots — pour le triple buffering (wave)

{ "name": "u", "type": "field", "numSlots": 3 }
// Wave du 2nd ordre : besoin de t-1, t, t+1
// numSlots: 3 → ParityManager gère le cycle modulo 3

WGSL et les macros auto-injectées

Quand tu appelles engine.getWgslHeader('collision'), le core injecte automatiquement des helpers pour toutes tes faces. Tu n'écris jamais d'index manuels.

Le header injecté — structure

// Automatiquement injecté avant ton code :

struct Uniforms {
  nx: u32,         // colonnes (sans ghost cells)
  ny: u32,         // lignes (sans ghost cells)
  strideRow: u32,  // nx + 2*ghostCells
  physicalNy: u32, // ny + 2*ghostCells
  t: u32,          // tick courant (frame)
  tick: u32,       // même chose (alias)
  p0..p7: f32,     // tes params (tau=p0, ux0=p1...)
  ...
}
@group(0) @binding(1) var<uniform> uniforms: Uniforms;

// Helpers d'indexation :
fn getIndex(x: u32, y: u32) -> u32
fn getIndex3D(x: u32, y: u32, z: u32) -> u32

// Pour chaque face déclarée — ex: face "phi" field :
fn read_phi(x: u32, y: u32) -> f32
fn write_phi(x: u32, y: u32, val: f32)

// Pour face multi-composante — ex: face "f" population :
fn read_f(x: u32, y: u32, d: u32) -> f32
fn write_f(x: u32, y: u32, d: u32, val: f32)

Correspondance params → WGSL

// Dans le manifest :
"params": { "tau": 0.55, "ux0": 0.08, "reynolds": 1000 }

// Dans le WGSL :
let viscosity = uniforms.p0;   // tau → p0 (ordre de déclaration)
let velocity  = uniforms.p1;   // ux0 → p1
let custom    = uniforms.p2;   // reynolds → p2
Les params sont mappés par ordre dans params, pas par nom. tau est toujours p0 parce qu'il est déclaré en premier.

Kernel minimal complet

const source = engine.getWgslHeader('compute') + `

@compute @workgroup_size(16, 16)
fn main(@builtin(global_invocation_id) id: vec3) {
  let x = id.x; let y = id.y;
  if (x >= uniforms.nx || y >= uniforms.ny) { return; }

  // Init au tick 0
  if (uniforms.tick == 0u) {
    write_phi(x, y, 1.0);
    return;
  }

  // Diffusion simple
  let val = (read_phi(x+1, y) + read_phi(x-1, y) + read_phi(x, y+1) + read_phi(x, y-1)) * 0.25;
  write_phi(x, y, val);
}`;

await engine.step({ 'compute': source });

Lire les résultats

Trois patterns selon ton besoin. Le premier ne coûte rien, le dernier coûte beaucoup — mais parfois nécessaire.

Pattern A — rendu direct (zéro coût)

Le buffer GPU est utilisé directement dans ton renderer. Aucun transfert CPU/GPU.

// Three.js : lier le buffer VRAM à un shader de rendu
const storageBuffer = engine.buffer.gpuBuffer;

const bindGroup = device.createBindGroup({
  layout: renderPipeline.getBindGroupLayout(0),
  entries: [{ binding: 0, resource: { buffer: storageBuffer } }]
});
// → ton vertex shader lit rho/ux/uy directement en VRAM

Pattern B — sync partielle (diagnostic, UI)

Rapatrie seulement les faces nécessaires. Ne bloque pas le render loop.

// À appeler hors du render loop (setTimeout, 1×/seconde)
async function updateHUD() {
  // Sync seulement 'rho' — pas toute la VRAM
  await engine.syncFacesToHost(['rho']);

  // Lecture SANS ghost cells (prête à itérer)
  const rho = engine.getInnerFaceData('chunk_0_0_0', 'rho');

  const avg = rho.reduce((a, b) => a + b, 0) / rho.length;
  document.getElementById('density').textContent = avg.toFixed(4);
}
setTimeout(updateHUD, 1000);

Pattern C — sync totale (export, debug)

// Sync tout (lent — uniquement hors boucle)
await engine.syncToHost();
const ux = engine.getInnerFaceData('chunk_0_0_0', 'ux');
const uy = engine.getInnerFaceData('chunk_0_0_0', 'uy');

// Export CSV ou envoi serveur
const csv = Array.from(ux).map((v,i) => `${v},${uy[i]}`).join('\n');

Réduction GPU (force, énergie totale)

// Somme d'une face directement sur GPU — sans readback
const totalEnergy = await engine.reduceField('rho', 'chunk_0_0_0');

// Forces aérodynamiques (méthode Momentum Exchange)
const [Fx, Fy] = await engine.reduceForces('chunk_0_0_0');

Trouver le chunkId

// Convention : chunk_x_y_z (position dans la grille de chunks)
// Grid 4×2 → chunks : chunk_0_0_0, chunk_1_0_0 ... chunk_3_1_0
// Pour une grille 1×1 (un seul chunk) : toujours chunk_0_0_0
const data = engine.getInnerFaceData('chunk_0_0_0', 'rho');
Pour une simulation 256×128 en 4×2 chunks → la grille physique fait 64×64 cellules par chunk. getInnerFaceData te retourne un Float32Array de 64×64 = 4096 valeurs par chunk.

La boucle principale

await engine.ready();

const kernels = {
  'compute': engine.getWgslHeader('compute') + computeSource
};

async function loop() {
  await engine.step(kernels, 1);
  requestAnimationFrame(loop);
}
loop();

setInterval(async () => {
  await engine.syncFacesToHost(['phi']);
  const data = engine.getInnerFaceData('chunk_0_0_0', 'phi');
  render(data);
}, 1000);
Les kernels tournent séquentiellement dans le même tick GPU. Les données sont immédiatement disponibles pour les kernels suivants dans le même step.
